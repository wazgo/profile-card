# 作りかけを引き継ぐ（100万円/月〜）— 発注前に渡す進行ガイド

> 目的：この1枚を発注前に相手に読んでもらい、「何を頼んでいるのか」「相手側が何をするのか」を握った状態で始める。
> profile-card のカードはこの内容を蒸留した短文。詳細はこのドキュメントが正本。
> 〔未決定〕タグの箇所は涼さんがまだ決めていない論点。

## 1. この依頼をする人は、どういう状態か（発注時のインサイト）

「作りかけがある。でも本番にできない」で詰まっている人。完成度も中身もバラバラなので、割合（7割）では括らない。**地点を問わず「残りで詰まっている」が共通項。**

### 今いちばん多い入口：AIで作った「見かけ上の完成品」

Claude Code 等をぶん回して、ローカルでは動く・見た目は完成しているものを持って来る。本人は楽しく作れて満足していて、**「サーバーに上げれば動く」と思っている。** だが中身は：

- ハードコード（値が埋め込まれていて変えられない）
- 見せかけの数字（モックで、実データが流れていない）
- バックエンドが不在（フロントだけ。保存・認証・処理が無い）

ここに、**性質の違う2つの問題**が同時にある。売り方・伝え方を分ける必要がある。

| | 問題1：上げても動かない | 問題2：上げられる形になっていない |
|---|---|---|
| 中身 | ハードコード／モック／バックエンド不在 | git管理が無茶苦茶、セキュリティが危険、デプロイ構成が無い |
| 相手の認識 | **見えていない。**「上げれば動く」と思い込んでいる | **薄々わかっている。**「やった方がいいと知っているが自分にはできない」 |
| リスク | 上げて動かず「**動かないじゃないか**」とこちらのせいにされる | （誤解は無い。素直に価値が伝わる） |
| 対応 | **見せて合意する**のが先。証拠を出してギャップを認識させる | 言えば頷く。**そのまま価値として提示できる** |

> 問題2は売りやすい（相手も必要性を知っている）。**危ないのは問題1。** 見かけ上できているぶん、本人の期待値は「あと少し」。実際は本番化に大きなギャップがある。このズレを**着手前に**埋めないと、後から「金を取ったのに動かない」になる。これを防ぐのが見立て（フェーズ0）の最大の仕事。

### 典型パターン

| # | 誰が | 何を持って来るか | 詰まっている所 |
|---|---|---|---|
| A | ノーコード/AIで自分で作った非エンジニア起業家 | 動くMVP・プロトタイプ | 認証・セキュリティ・データ・スケール。本番品質の壁 |
| B | 外注/フリーランスが抜けた事業者 | 中途半端なコードベース | 引き継げる人がいない。状態が分からない |
| C | 社内の片手間で作られたプロジェクト | 業務の傍らで作られたもの | 本番化の専門性が社内にない |
| D | PoC/社内検証が通った会社 | 検証済みの仮説＋簡易実装 | 本番化する手と判断が社内に足りない |

### 共通する心理（ここを外すと刺さらない）

- **「ここまで来たのに、進まない」焦り。** 動いているのに本番にできないもどかしさ。
- **コードの状態への不安。** 「これ使えるの？作り直し？」が言語化できていない。
- **何が足りないか自体が分からない。** 「本番化に何が要るか」が分からないまま見積もりを取ろうとしている。
- **予算が青天井になる恐怖。** 「言い値で延々取られる」過去の外注失敗の記憶。
- **意思決定者本人であることが多い。** だから判断は速いが、忙しくて手が回らない。

### よくある誤解 → 本当に必要なこと

| 相手が思っていること | 実際に必要なこと |
|---|---|
| 「残りの実装をやってくれる人」が欲しい | 「**どこまでやれば本番か**」を一緒に決めて、その判断ごと引き取る人 |
| 「100万払えば全部やってくれる」 | 相手も**判断と情報を出す**。前進は共同作業 |
| 「完成日を約束してほしい」 | 状況で変わるので、**毎月の前進**を約束する形の方が双方守られる |

> この誤解のまま受けると「金払ったのに進まない業者」になる。だから**発注前にこのギャップを埋める**のがこのガイドの役目。

### 発注に至る瞬間（トリガー）

作りかけは、抱えたまま動かないのが常態。**何かのトリガーで初めて外に出す。** 共通項は感情ではなく、**「自分の手では、ここから先に進めない」と向き合う瞬間。** ただし感情の色は3つに分かれ、**売り方が変わる。**

| 型 | きっかけ | 心理 | 刺さる伝え方 |
|---|---|---|---|
| **楽観型（最頻・AIケース）** | ローカルで動いた・見た目は完成 | 焦ってない。「いけそう／あと少し／上げれば動く」 | **速さより先に "気づかせる"（A）。** いきなり速さを売ると「急いでないし」になる |
| **焦り型** | 期限が来た／見せたら壊れた／人が抜けた | 切迫している | **「速い・固定額・地図が出る」** が刺さる |
| **モヤモヤ型** | 「本当にこれでいいのか」と不安が芽生えた | 確信が持てない | 状態を可視化して**判断材料を渡す** |

> 注意：最頻の入口（AIで楽しく作れた人）は**焦っていない。むしろ前向き・楽観的。** だから「焦り前提」で速さを押すと外す。彼らにはまず見立てでギャップに気づいてもらうのが先。焦り型と楽観型を取り違えないこと。

### 納得して発注してもらう順番

納得＝「自分のものの本当の状態」と「ここから本番までの距離」が腑に落ちること。**営業トークではなく、見立てそのものが説得になる。**

1. **無料の短い対話**で「上げたら動くと思ってます？」を確認 → ほぼ「思っている」
2. **見立て（有償）**で実物を見せながらギャップを指し示す → 「これは自分じゃ無理だ」と**相手が自分で気づく**
3. その地図を持って100万の判断 → 丸投げではなく、**何にお金を払うかが見えた状態**で発注

> 説得しない。**気づかせる。** こちらが「ヤバいですよ」と言うほど営業臭くなる。実物とギャップマップに語らせる方が、相手は納得して・自分の意思で発注する。

## 2. 売っているもの（再定義）

**実装はする。むしろ中核で書く。** 手を動かさない"判断だけの人"ではない。フリーランスとの差は、手を動かすだけでなく**判断も込みで進める**こと（判断だけでも実装だけでもない）。

ただし——**売っているのは「実装の着地（＝完成・本番化の達成）」ではない。「実装と判断を、毎月前に進めること」。**

着地を成果物にすると困る：

- 本番化が相手の状況で**不能でも、着地保証だと対価が取れない**（リスクが全部こちら持ちになる）
- **スコープが膨らむと着地が遠のく**＝終わらない
- 「完成日はいつ」の請負になり、**月額の意味が消える**

だから成果の単位は **「前進」**。完成ではなく、毎月確実に前に進むこと。これが「約束するのは毎月の前進」の根拠であり、リスクヘッジの核。

- 0→1（まず動くものをつくる・55万）とは別。これは **作りかけ → 運用に耐える状態**。
- 中核に居続ける（プロダクト推進パートナー・300万）とも別。これは **本番化フェーズ限定の月額**。区切りがある。

## 3. 進行

### フェーズ0：見立て（有償・固定・1〜2週間）　〔未決定：独立メニューにするか／金額〕

いきなり月額の実装に入らない。最初に短期・固定額の診断を必ず通す。

**この見立ての最大の仕事は「見かけ上の完成度 ≠ 本番の完成度」を、証拠を見せて合意すること。** 後から「動かないじゃないか」と言われないための装置。

やること：
1. 作りかけの状態を実際に見て、**「見える形」と「本番に必要なもの」のギャップを可視化する**
   - どこがハードコード／モックか、バックエンドの有無を**具体的に指し示す**（「ここの数字は埋め込みで、実データは流れていません」）
   - git・デプロイ・セキュリティの状態を見せる（問題2は相手も納得しやすい）
2. 既存ビジネスのヒアリング（事業・収益・既存サービスとの関係）
3. **本番の定義（DoD）を合意** — 認証・セキュリティ・運用・データ・法務、どこまでやれば本番か
4. やりたいこと全部出し → **本番化に要る／後でいい／別案件** に仕分け

アウトプット：**「見かけ vs 本番」ギャップマップ＋本番化マップ＋初月の着手対象**（＝後で揉めないための、合意済みの証拠）

狙い：
- **問題1（見えないズレ）を着手前に共有の事実に変える**＝「動かないじゃないか」を構造的に防ぐ
- 既存ビジネスのヒアリングがここで自然に回収される
- 「金払えば全部」「意思決定者が出てこない」「丸投げ」を**この入口で落とす**＝フィルタ
- 相手も「触れるもの＋地図」で判断が変わる（55万メニューと同じ思想）

> 伝え方のコツ：相手の作ったものを否定しない。**「デモとしては完成しています。本番にするにはここが要ります」**と立てる。本人は楽しく作れた成功体験があるので、そこを壊さずギャップだけを見せる。

### フェーズ1：月次で前に進める（100万円/月〜）

売っているのは「毎月の前進」。だからリズムで回す。

- **月初**：マップから「今月前に進める 1〜3個」を合意
- **週次**：短い同期（進捗・詰まり・判断待ちの確認）
- **月末**：レビュー＝今月やった／次月やる／マップの残り → **双方が継続を判断**（いつでも降りられる＝リスクヘッジ）

まずは1〜3ヶ月を目安に、手応えがあればその先も。

### スコープが膨らんだとき（歓迎して、握り直す）

膨らむのは悪じゃない。問題は膨らんだまま今月に押し込まれること。

- **本番化と新規機能を分ける。** 本番化＝今あるものを運用に耐える状態に。新しくやりたいことは「別線」。
- 本質的に大きくなったら **300万/月（推進パートナー）へ格上げ**＝アップセル導線。
- **追加課金ではなく優先順位の引き直しで吸収。** マップを更新して順番を変える。金額で殴り合わない。

## 4. うまくいく条件（＝お客様にお願いすること）

ここが期待値合わせの核心。「相手が何もしなくても毎月100万ぶん進む」と思われた瞬間にズレる。だから**相手側の義務を発注前に握る。**

1. **意思決定の窓口を1人決め、月次レビューに出る**
2. **判断待ちを溜めない** — こちらが投げた確認・判断に〔未決定：〇営業日〕で返す
3. **情報・権限を開く** — 既存コード、データ、アカウント、外部サービスのアクセス
4. **膨らんだら黙って増やさず、マップを一緒に引き直す**
5. **「毎月の前進」を成果とみなす**（完成日を握らない）

> **期待値通り＝この5つが回っている状態。** 回らなければ前進は鈍るが、それは相手要因で対価は変わらない。これを先に言葉にしておくのがリスクヘッジの本体。

---

## 5. 約束すること／しないこと

- **約束する：** 毎月、本番化マップに沿って前に進めること。判断を引き受けること。動いた分の対価に見合う前進。
- **約束しない：** 特定の完成日。本番化が相手の状況で不能な場合でも結果は保証しない（ただし**動いた月の対価は出る**＝時給制ではなくコミットの対価）。

## 6. 見極めと売り方

### A. 問題1の「見せ方」＝気づかせる手口

説得ではなく、**本人に発見させる**のが一番効く。見立ての中で、実物を一緒に見ながら問いを置く：

1. 「この数字、どこから来てます？」 → ハードコード／モックを**本人に見つけさせる**
2. 「保存したらどうなります？」「ログインは誰が管理してます？」 → バックエンド不在を体感させる
3. デプロイ構成・環境変数・秘密情報の置き方を見る → 問題2（セキュリティ）を可視化
4. 結果を **「見かけ vs 本番」ギャップマップ1枚**に。✅できてる／⚠️見かけだけ／❌無い で色分け

- 全部を本番相当で再現すると工数が出るので、**代表的な1点だけ実演**して残りはマップで示す
- トーンは終始「ここまで動かせたのはすごい。本番にするにはこの3つが要る」＝**否定しない**
- このマップが**「動かないじゃないか」を防ぐ合意の記録**になる（後で揉めない）

### B. 断るべき相手（アンチパターン）

受けない線を先に決めると、受ける相手の質が上がる。見立て（有償ゲート）がこれらをほぼ自動で落とす。**落ちることを恐れない。**

- **丸投げしたい人** — 意思決定者が会話に出てこない／判断を全部こちらに投げる
- **値切り前提** — 「もっと安く」「フリーランスならいくら」＝土俵が違う
- **「とりあえず安く動けばいい」** — 本番品質に金を払う気がない（55万デモの客か、別の客）
- **見立てをタダでやらせようとする** — 地図だけ抜きに来る人
- **目的・既存事業を説明できない／コロコロ変わる** — 本番化の的が定まらない
- **過去の外注を全部"相手のせい"にする人** — 次もこちらのせいにする

### C. 競合・代替手段との差

相手が他に取りうる手と、その落とし穴。これが「なぜ見立て＋引き継ぎか」の答えになる。

| 相手の代替手段 | 落とし穴 |
|---|---|
| 安いフリーランスに残りを頼む | コードの状態を読めない人だと泥沼。手は動くが**判断はしてくれない** |
| 自分でAIで粘る | セキュリティ・運用・判断は**ぶん回しの外側**。AIに丸投げできない領域 |
| 諦めて作り直し発注 | 既存資産が無駄に。時間と金が倍 |
| フルタイムエンジニア採用 | 採用に数ヶ月、1人で本番化の全領域は無理、コストも月100万級 |

→ 差別化＝**「作りかけの状態を読んで、判断ごと引き取る」**。手だけでもゼロからでも採用でもない、"今ある地点から本番への最短の判断"。

### D. 予算と意思決定構造（誰がどう100万/月を正当化するか）

| 相手 | 財布 | 正当化のロジック | 見立て(25万)の役割 |
|---|---|---|---|
| 個人・ソロ起業家 | 自腹／調達 | 100万/月は重い。**1〜3ヶ月の区切り＋前進が見える**で納得 | 小さく試す入口 |
| スタートアップ（調達済み） | バーンレート | 「採用より速い・密度」。投資家への期限がトリガー | CTO不在の判断補完 |
| 中小・事業会社 | 既存事業/新規予算 | ROIより**「止まってるものが動き出す」**が決裁理由 | 稟議を通す材料（地図） |

- 共通項：**「採用・育成の時間と不確実性を、毎月の前進で買う」**。フルタイム1名分の単純比較に乗らない（料金思想と一致）。
- 見立て(25万)は「社内で100万を通すための、小さな最初の一歩」としても機能する。
